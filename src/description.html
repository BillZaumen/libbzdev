The BZDev Class Library

The BZDev class library is partitioned into 8 modules as shown in the following diagram (also shown on the OVERVIEW page):

All of these modules are dependent on the module org.bzdev.base, but only some dependencies are shown explicitly to reduce clutter: just enough so that the diagram will show at least an indirect dependency. Two modules, org.bzdev.dmethods and org.bzdev.parmproc, are not shown as these are used only for annotation processing.

The module org.bzdev.base

The module org.bzdev.base provides basic operations. The following diagram shows the packages this module contains:

The packages in this module provide the following:
  • org.bzdev.lang provides support for dynamic methods - methods in which dispatching based on argument types is performed at run time instead of compile time. It also provides an interface named Callable that is analogous to java.lang.Runnable, but is intended for code that should be called rather than run in new thread. A Callable has a method named call() that takes no arguments and does not return a value or throw a checked exception. Related interfaces all the 'call' method to return a value, declare exceptions, and declare arguments. The class MathOps is a "utility class" that provides simple operations involving Java's primitive types and is effectively a trivial extension of the Java package java.lang.Math.
  • org.bzdev.scripting allows scripts to be run in a sandbox while normal Java code is not.
  • org.bzdev.io provides some IO-related classes. These include a Writer class that can write to an Appendable, some classes that allow opening of files or directories to be specified before a security manager restricts access and actually opened afterwards, and classes for reading and writing files in some specialized formats.
  • org.bzdev.net and org.bzdev.protocols provide networking classes:
    • org.bzdev.net contains classes to augment the java.net package.
    • org.bzdev.protocols contains additional protocol handlers.
  • org.bzdev.util provides some utility classes.
  • org.bzdev.util.units provides classes for converting from one set of units to another.

The module org.bzdev.math

The module org.bzdev.math provides special functions, various numerical algorithms, random variables, and statistics. The following diagram shows the packages this module contains:

The packages in this module provide the following:
  • org.bzdev.math provides some special functions and numerical-analysis algorithms. These are useful in general, but are also needed to support the org.bzdev.geom package.
  • org.bzdev.math.stats provides basic support for statistics: a few statistical tests and statistical distributions. It also includes classes that use numerically stable algorithms for computing means and standard deviations.
  • org.bzdev.math.rv provides a collection of classes implementing random variables.
  • org.bzdev.math.spi contains service-provider interfaces for the org.bzdev.math package.
  • org.bzdev.providers.math.fft contains a default service-provider for an FFT (Fast Fourier Transform).

The module org.bzdev.graphics

The module org.bzdev.graphics provides graphs and IO operatios involving geometry, graphics-based IO operations, graphs, and packages that support factory classes for simulations, animations, etc. The following diagram shows the packages this module contains:

The packages in this module provide the following:
  • org.bzdev.graph provides a graph-plotting package stressing ease of use rather than a large number of features. This package is also used to support the org.bzdev.anim2d package described below.
  • org.bzdev.geom provides information about paths and subclasses of java.awt.geom.Path2D to support Bézier splines.
  • org.bzdev.gio provides a service for creating image files, including postscript files, while using an API similar to that for a BufferedImage. An SPI (Service Provider Interface) allows additional image types to be created. If the Apache Batik SVG jar files are installed with batik.jar on the class path, the creation of SVG files will be enabled. This package also provides a convenient interface for printing.
  • org.bzdev.imageio provides a class that can scale images and a class to obtain MIME type information about image formats.
  • org.bzdev.obnaming.misc contains classes that can be used with named-object factory annotations, handling colors, strokes, and fonts.

The module org.bzdev.desktop

The module org.bzdev.base provides graphics and desktop operations. The following diagram shows the packages this module contains:

The packages in this module provide the following:
  • org.bzdev.swing provides swing components.
  • org.bzdev.swing.io contains a Reader that can extract text from a Document (the class Reader is in the java.io package and the class Document is in javax.swing.text).
  • org.bzdev.swing.keys contains a class that can look up virtual keys by name. This is useful in conjunction with a GUI.
  • org.bzdev.swing.proxyconf allows proxies to be configured.
  • org.bzdev.swing.text provides thread-safe versions of some classes in javax.swing.text, plus some classes that are useful for validating user input and limiting the characters that can be entered into a text component.

The module org.bzdev.devqsim

The module org.bzdev.devqsim provides discrete event and queuing simulation. The folloowing diagram shows the packages this module contains:

The package

  • org.bzdev.devqsim provides a scriptable discrete event queue simulation library with support for queues of various types. Users of this library will define instances of the interface org.bzdev.Callable to create code that can be run at specified simulation times via a method call. For instances of java.lang.runnable, a thread will be started at specified simulation times. Threads (instances of org.bzdev.devqsim.TaskThread) can also be run at specific simulation times and may pause when desired. The library also supports various types of queues.
  • org.bzdev.devqsim.rv provides random variables that are 'named objects' that a simulation can recognize, with factory methods for creating them.

The module org.bzdev.drama

The module org.bzdev.drama provides discrete event simulations using a message-passing paradigm. The following diagram shows the packages this module contains:

The package

  • org.bzdev.drama (whose classes extend the classes in org.bzdev.drama.generic) provides simulations in which entities called actors communicate by passing messages to each other. The drama package is really a trivial extension of the org.bzdev.drama.generic package. For the most part, it just provides concrete names and eliminates the need for most of the type parameters used by org.bzdev.drama.generic.
  • org.bzdev.drama.generic provides the basic operations provided by the org.bzdev.drama package, but uses generic types so that multiple "flavors" of drama-like simulation libraries can be created. For example, in some application domains, the term Agent is preferable to Actor, and each agent may have some additional methods beyond those provided by the corresponding class in org.bzdev.drama.generic.
  • org.bzdev.drama.common contains classes that are useful for all flavors of drama-like simulations.

The module org.bzdev.ejws

The module org.bzdev.base provides an embedded web server based on the Java package com.sun.net.httpserver that is part of the standard Java release. The following diagram shows the packages this module contains:

These packages are described below:

  • org.bzdev.ejws simplifies configuring an HttpServer or HttpsServer provided by the com.sun.net.httpserver packages. It includes an HttpHandler class (FileHandler) that uses classes defined in the package org.bzdev.ejws.map to map URLs into objects that can be used to satisfy an HTTP request.
  • org.bzdev.ejws.map contains a collection of classes that map a URL to resource and allow that resource to be read.

The remaining modules

The remaining modules are ones with one package (sometimes a substantial package) per module. The following diagram shows these modules and their packages:

  • org.bzdev.anim2d provides support for simple, 2D animations based on the org.bzdev.devqsim package, the org.bzdev.graph package, and the org.bzdev.geom package.
  • org.bzdev.obnaming provides a service in which an object namer allows named objects to be looked up by name. Generics allow the types for named objects and object namers to be specified, and a framework for creating factories to generate named objects is also provided. An annotation processor simplifies the creation of factories. Some classes useful for building factories when using annotations are in the package org.bzdev.obnaming.misc, and are used in the org.bzdev.anim2d package (see below).
  • org.bzdev.p3d provides support for 3D printing. This package provides classes that allow a 3D model to be shown in an animation (using the org.bzdev.anim2d package) that rotates an object so it can be viewed from multiple angles. It can also use classes in the org.bzdev.geom package for generating complex shapes, and it can use the org.bzdev.graph class to create images.

Compilation

The following instructions are appropriate for Debian Linux and other distributions that put libraries in the same locations. The procedure depends on whether Java modules are used or not.

To compile without modules and without annotation processors, assuming the compiled classes will be put in the directory classes 

      javac -d classes -classpath /usr/share/java/libbzdev.jar \
             JAVA_FILE...
To compile without modules and with annotation processors, add a directory tmpsrc to hold the generated source-code files: 
      javac -d classes -classpath /usr/share/java/libbzdev.jar \
             -s tmpsrc JAVA_FILE...
In both cases JAVA_FILE should be a path whose components reflect the package hierarchy. For example, the class com.foo.pkg.Foo will be in the file com/foo/pkg/Foo.java.

To compile with modules and without annotations, the source file should be in subdirectories of a directory traditionally named src. These subdirectories should have names matching a module name (for example, com.foo.mod). The compiled code goes into similar subdirectories of a directory named mods: 

      javac -d mods/com.foo.mod -p /usr/share/bzdev JAVA_FILE...
In this example, src/com.foo.mod must contain a file named module-info.java: 
      module com.foo.mod {
        ....
      }
In addition, a Java source file for the class com.foo.pkg.Foo.java in the module com.foo.mod must be in the directory src/com.foo.mod/com/foo/pkg/Foo.java, with all modules and packages being compiled following this pattern.

To compile with modules and with annotation, the javac command above should be changed to 

      javac -d mods/com.foo.mod -p /usr/share/bzdev \
          -s tmpsrc/com.foo.mod \
          --processor-module-path /usr/share/bzdev \
          src/com.foo.mod/module-info.java JAVA_FILE...

When javadoc is used to generate API documentation, the command for the previous example without modules when annotation processing is not used is: 

      javadoc -d APIDIR -sourcepath . \
          -classpath classes JAVA_FILE...
When annotation processing is used, this command becomes 
      javadoc -d APIDIR -sourcpath .:tmpsrc \
          -classpath classes JAVA_FILE...
where APIDIR is the directory that will contain the API documentation. Similarly, when modules are used without annotation processing, the javadoc command is 
      javadoc -d APIDIR --module-path JARDIR \
          --add-modules com.foo.mod \
          --module com.foo.mod JAVA_FILE...
where JARDIR is a directory that contains the JAR files for the modules. When annotations are used, this becomes 
      javadoc -d APIDIR --module-path JARDIR \
          --module-source-path src:tmpsrc \
          --add-modules com.foo.mod \
          --module com.foo.mod JAVA_FILE...

There is a minor detail to watch: the options --module-path, module-source-path, and --add-modules allow multiple objects to be listed, separated by a colon. The --module option requires that the module names be separated by a comma.

Finally, when named object factories are used, the factories can describe their parameters. The program lsnof can generate documentation describing a factory's parameters. When modules are not used the command is 

      lsnof -d APIDIR --classpathCodebase JARFILE CLASSNAME...
and when modules are used, the command is 
      lsnof -d APIDIR --module-path JARDIR CLASSNAME...
While fully-qualified class names can be provided explicitly, lsnof also supports a wildcard syntax described in the lsnof man page and elsewhere. In both cases, the documentation will appear in a subdirectory of APIDIR named factories-api (the hyphen is used to prevent collisions with Java package names). If a factory's fully qualified class name is org.foo.pkg.Factory, one can place the parameter documentation in an IFRAME by using an IFRAME whose source attribute is 
   SRC="{@docRoot}/factories-api/org/foo.pkg/Factory.html"
The javadoc program will replace {@docRoot} with the API directory (APIDIR in this example).

Libraries

The JAR files for the libbzdev modules should be available in a single directory. On Debian Linux systems, unless modified by a package maintainer, the files are in /usr/share/java. Typically an application will set up symbolic links to these files (there are some files in /usr/share/java that interact pooly with Java modules, making it problematic to put /usr/share/java on a module path). The JAR files for the lsnof and scrunner commands are located in /usr/share/bzdev and when these are installed (the Debian package is libbzdev-util), symbolic links to /usr/share/java from /usr/share/bzdev will be set up as well.

Normally, just setting the module path to include /usr/share/bzdev is sufficient. However, if an application has to run on a device with tight memory/storage constraints, or if jlink is used to create a custom JVM for a specific application, it may be desirable to include a subset of these Jar files.
